### 通用执行规则

1. 每一轮只完成一个明确、可验证的工作闭环。

2. 每轮开始前明确本轮的允许范围和禁止范围，不得擅自扩展任务。

3. 完成当前目标并通过验证后立即停止，不得自动开始下一阶段。

4. 不允许多个 Agent、模型或代码助手同时修改同一个代码仓库，避免文件冲突和实现方向不一致。

5. 原始需求应保存在项目文件中，后续任务只引用该文件，不要反复将完整需求粘贴到对话中。

6. 已经确认的技术架构、数据库结构和目录设计不得在每一轮重新设计。只有发现明确缺陷或需求发生变化时才允许调整。

7. 修改架构前必须说明修改原因、影响范围和迁移方式，不得无理由推翻已有实现。

8. 测试失败时，只修复与当前阶段直接相关的问题，不得借机重构无关模块或增加新功能。

9. 每轮修改文件的范围应尽量集中，避免一次跨越前端、后端、数据库、部署和测试等过多模块。

10. 外部服务、数据源或第三方 API 应逐个接入。一个接入完成并测试通过后，再开始下一个。

11. 优先完成最小可运行闭环，再逐步补充边界处理、异常处理和扩展能力。

12. UI 精修、动画和视觉优化应放在数据结构、接口和核心交互稳定之后进行。

13. 不要在核心业务逻辑尚未完成时，大量投入时间调整颜色、间距、阴影或动画。

14. 复杂架构设计、安全问题、疑难错误和跨模块决策，使用推理能力更强的模型或更高推理模式。

15. 文件创建、格式整理、普通 CRUD、样式微调、文档补充和重复性代码，可以使用速度更快或成本更低的模型。

16. 切换模型时，必须以项目文件、代码仓库和进度文档作为事实来源，不依赖上一模型未保存的对话记忆。

17. 每轮任务开始前先读取：

* 原始需求
* 实施计划
* 当前进度
* 相关代码
* 最近的测试结果

18. 不得假设某项功能已经完成。必须通过实际文件、数据库迁移、接口响应或测试结果确认。

19. 不得伪造命令执行结果、测试通过结果、构建结果、API 响应或运行截图。

20. 无法执行某项验证时，必须明确标注“未验证”，并说明原因。

21. 每轮结束前必须运行与当前修改相关的验证，例如：

* 类型检查
* 代码检查
* 单元测试
* 集成测试
* 数据库迁移
* 生产构建
* 端到端测试

22. 不要求每一轮都运行所有测试，但必须运行足以验证当前修改的测试。

23. 每轮结束后更新项目进度文档，记录：

* 已完成内容
* 修改的关键文件
* 执行的命令
* 测试结果
* 已知问题
* 未完成内容
* 下一阶段建议

24. 不得为了让页面或项目显得更完整而添加需求之外的功能、按钮、字段、页面或假数据。

25. 当需求不明确时，优先选择最小实现，不自行扩展产品范围。

26. 当多个实现方案都可行时，优先选择依赖更少、结构更清晰、后期更容易维护的方案。

27. 不要进行与当前目标无关的大规模重构。

28. 不要为了统一代码风格而修改大量无关文件。

29. 不要在一个任务中同时进行功能开发、框架升级、依赖全面更新和架构迁移。

30. 每次升级框架或依赖时，应单独执行，并完成兼容性验证。

31. 对高风险操作建立单独阶段，包括：

* 数据库结构变更
* 数据迁移
* 文件删除
* 权限修改
* 安全逻辑
* 部署配置
* 第三方 API 更换

32. 删除文件、数据或已有功能前，必须确认其确实不再被使用。

33. 所有核心功能必须有明确的完成标准，不能以“代码已经写了”作为完成依据。

34. 一个阶段只有在以下条件满足时才算完成：

* 功能真实可用
* 关键错误已处理
* 相关测试通过
* 文档已更新
* 没有超出需求范围
* 没有遗留未说明的核心 TODO

35. 优先保证正确性、稳定性和可维护性，再考虑开发速度和代码数量。
