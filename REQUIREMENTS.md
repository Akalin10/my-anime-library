你是一名资深产品经理、UI/UX 设计师、前端工程师、后端工程师和数据库工程师。

请从零开始设计并实现一个完整、可运行、可维护的私人动漫收藏网站。

项目暂定名称：

「我的动漫库」

这是一个仅供个人使用的私人动漫收藏网站。

它不是社交平台。
它不是动漫社区。
它不是评分网站。
它不是追番进度工具。
它不是数据统计后台。
它不是 MyAnimeList、AniList 或 Bangumi 的复制品。

这个网站最核心的用途只有三个：

1. 搜索并导入动漫资料。
2. 查看自己收藏的动漫海报和资料。
3. 标记一部动漫是“在看”还是“已看完”。

请严格按照本需求实现。

不得自行增加任何未明确要求的功能。

当你认为某个功能“可能有用”时，不要擅自添加。
只有本需求明确列出的功能才允许出现。

==================================================
一、项目定位
==================================================

这是一个单用户、本地优先、私人使用的动漫资料收藏网站。

用户不需要记录评分、笔记、观看集数、观看日期或观看时长。

用户只关心：

- 我正在看哪些动漫。
- 我已经看完哪些动漫。
- 某个动漫系列还包含哪些季度、OVA、OAD、特别篇和剧场版。
- 自动获取动漫的海报、简介和基础资料。
- 在不满意默认封面时更换成自己上传的图片。

整个产品应当像一个安静、克制、精致的私人动漫图书馆。

它不应该像后台管理系统。
它不应该像数据仪表盘。
它不应该像传统动漫社区。
它不应该有密集的数据和大量按钮。

==================================================

==================================================
三、核心状态
==================================================

动漫状态只能有以下两个：

1. 在看
2. 已看完

数据库内部可以使用：

WATCHING
COMPLETED

界面中必须显示中文：

在看
已看完

禁止增加以下状态：

- 想看
- 计划观看
- 暂停
- 搁置
- 弃看
- 重看
- 未开始
- 追更中
- 已收藏但未观看

每一部导入的动漫必须选择“在看”或“已看完”。

导入时默认状态可以设置为“在看”，但必须允许用户在导入前切换。

左侧导航栏保持简洁。

顶部显示：

我的动漫库

MY ANIME LIBRARY

导航项目只允许有：

- 全部
- 在看
- 已看完
- 设置

每个导航项目可以显示数量，例如：

全部 120
在看 8
已看完 112

不要增加：

- 排行

6.2 顶部区域

- 网格视图按钮

搜索框用于搜索本地动漫库。

输入内容后，应当在当前已经导入的动漫中匹配：

6.3 状态筛选

不要使用高饱和度彩色标签。

不要使用大型胶囊按钮。

不要使用红绿蓝三种强烈状态色。

6.4 排序

6.5 海报网格

平板端显示 3 至 4 张。

手机端显示 2 张。

每个海报卡片平时只显示：

- 海报图片
- 动漫标题

禁止在卡片上显示：

- 星星

状态文字使用：

在看
已看完

鼠标悬停海报时：

- 海报轻微放大
- 卡片轻微上浮
- 可以出现一层很淡的遮罩
- 可以显示“查看详情”

==================================================

==================================================

点击海报后，使用独立模态窗口展示详情。

模态窗口必须：

- 居中显示
- 使用半透明背景遮罩
- 背景可以轻微模糊
- 不改变首页布局
- 不占用首页底部空间
- 不跳转页面
- 可以使用右上角关闭按钮关闭
- 可以按 ESC 关闭
- 点击遮罩区域可以关闭
- 模态窗口内容过长时，只允许模态窗口内部滚动
- 打开模态窗口后禁止页面背景滚动

桌面端模态窗口建议宽度：

900px 至 1100px

最大高度：

视口高度的 85% 至 90%

布局：

左侧海报
右侧资料

左侧海报约占 35%。

右侧资料约占 65%。

- 动画类型
- 总集数

动画类型可以是：

不要显示人气。

7.2 状态切换
------------------------------------------

详情模态窗口提供状态切换：

在看
已看完

推荐使用两个简洁选项，或一个下拉选择器。

状态切换后立即保存。

保存成功后无需出现大型提示。

可以使用很小的轻提示：

已保存

7.3 相关作品

相关作品可以按以下类型分组：

- 正传
- 前传

- OVA / OAD
- 剧场版
- 特别篇
- 总集篇

- 标题
- 年份
- 类型

已经导入时显示：

已添加

未导入时显示：

未添加

点击未导入的相关作品，可以打开简单的导入确认界面。

不要自动导入用户没有选择的相关作品。

7.4 删除条目
------------------------------------------

删除前显示确认窗口：

确定要从动漫库中删除《作品名称》吗？

删除只删除本地收藏数据和本地自定义封面。

不要操作外部数据源。

==================================================
八、搜索并导入动漫
==================================================

用户点击“添加动漫”后，打开搜索导入界面。

可以使用：

- 独立全屏模态窗口
- 覆盖当前页面的大型对话框

不要跳转到复杂的后台页面。

------------------------------------------
8.1 搜索输入
------------------------------------------

顶部显示一个明显的搜索框。

占位文字：

输入动漫名称，例如：进击的巨人

用户可以输入：

- 中文名
- 日文名
- 英文名
- 别名

输入后点击搜索，或停止输入一段时间后自动搜索。

自动搜索应当使用防抖机制。

建议防抖时间：

400ms 至 600ms

搜索过程中显示简洁的加载状态。

不要显示假结果。

不要在 API 还没有返回时使用预设动漫填充。

------------------------------------------
8.2 搜索结果目标
------------------------------------------

例如用户搜索：

进击的巨人

搜索结果应尽可能包含：

- 进击的巨人 第一季
- 进击的巨人 第二季
- 进击的巨人 第三季
- 进击的巨人 第三季 Part.2
- 进击的巨人 最终季
- 进击的巨人 最终季 Part.2
- 进击的巨人 最终季 完结篇
- OVA
- OAD
- 特别篇
- 总集篇
- 剧场版
- 外传
- 衍生动画

但禁止根据名称自行编造不存在的季度或作品。

必须使用数据源返回的真实条目和真实关联关系。

------------------------------------------
8.3 搜索结果分组
------------------------------------------

搜索结果尽量按照作品类型分组：

正传
OVA / OAD
剧场版
特别篇
总集篇
外传 / 衍生
其他结果

如果数据源无法确定作品关系，不要强行分类。

无法确定时统一放到：

其他搜索结果

不要根据标题中出现“2”“Season”“Part”就武断判断关系。

优先使用数据源明确提供的：

- prequel
- sequel
- parent
- side story
- spin-off
- adaptation
- alternative
- compilation
- special

等关系字段。

------------------------------------------
8.4 搜索结果卡片
------------------------------------------

每个搜索结果显示：

- 小型海报
- 中文标题
- 原始标题
- 年份
- 类型
- 总集数
- 数据来源
- 是否已经导入
- 选择框

搜索结果不要显示：

- 评分
- 排名
- 热度
- 用户数
- 收藏人数
- 评论数量

用户可以：

- 单独选择一个条目
- 同时选择多个条目
- 取消选择
- 导入选中的条目

不要默认全选。

不要自动导入所有相关作品。

------------------------------------------
8.5 导入前状态
------------------------------------------

导入前允许为选中的作品设置状态：

- 在看
- 已看完

默认状态：

在看

用户可以统一设置全部选中作品的状态。

也可以在每个结果上单独设置状态。

------------------------------------------
8.6 导入过程
------------------------------------------

导入时应当：

1. 获取完整动漫资料。
2. 获取最佳可用海报。
3. 获取相关作品关系。
4. 将资料标准化。
5. 检查是否已经导入。
6. 避免重复创建。
7. 将海报下载到本地。
8. 保存数据库记录。
9. 返回导入结果。

导入过程中显示真实状态，例如：

正在获取资料……
正在下载封面……
正在保存……
导入完成

出现错误时明确告诉用户是哪一个作品失败。

批量导入时，一个条目失败不应该导致全部失败。

导入完成后显示：

成功导入 X 部作品
失败 X 部作品

不得显示虚假的成功提示。

==================================================
九、外部动漫数据源
==================================================

采用可扩展的数据源适配器架构。

初始支持的数据源可以包括：

- Bangumi
- AniList
- TMDB

其中：

Bangumi 和 AniList 优先用于动画条目和系列关系。
TMDB 可以作为电影、剧场版、海报或背景资料的补充来源。

实现前必须阅读这些数据源当前的官方 API 文档。

不要凭记忆猜测接口地址。

不要使用已经废弃的接口。

不要把 API 密钥写入前端代码。

不要把密钥提交到代码仓库。

所有需要密钥的配置必须通过环境变量提供。

------------------------------------------
9.1 数据源适配器
------------------------------------------

每个数据源实现统一接口，例如：

searchAnime(query)
getAnimeDetail(sourceId)
getAnimeRelations(sourceId)
getPosterCandidates(sourceId)

标准化后的统一数据结构至少包含：

- source
- sourceId
- titleChinese
- titleNative
- titleEnglish
- aliases
- year
- mediaType
- episodeCount
- studio
- synopsis
- posterUrl
- relations

不同数据源字段缺失时允许为空。

不得编造缺失数据。

------------------------------------------
9.2 多数据源搜索
------------------------------------------

搜索可以并行请求已经启用的数据源。

搜索结果需要去重。

去重时可以综合参考：

- 标准化标题
- 原始标题
- 上映年份
- 动画类型
- 集数
- 数据源外部 ID
- 明确的跨平台 ID

不要仅凭中文标题相同就合并。

无法确认是同一条目时，保留为独立结果。

合并结果时必须保留所有来源信息。

------------------------------------------
9.3 数据源故障
------------------------------------------

某个数据源不可用时：

- 不要让整个搜索界面崩溃
- 显示该数据源请求失败
- 其他数据源继续工作
- 提供重试按钮
- 不要显示空白错误页
- 不要使用假数据代替错误结果

------------------------------------------
9.4 请求缓存
------------------------------------------

对搜索和详情请求进行适度缓存。

避免在短时间内重复请求相同内容。

缓存可以设置合理过期时间。

但缓存不得导致用户一直看到旧搜索结果。

==================================================
十、封面处理
==================================================

海报是整个网站的核心视觉元素。

每部动漫必须支持：

1. 数据源默认封面
2. 用户上传本地封面
3. 用户通过图片 URL 添加封面
4. 恢复数据源默认封面

------------------------------------------
10.1 默认封面
------------------------------------------

导入动漫时，优先获取质量较高、竖版、适合 2:3 展示的海报。

保存时尽量将远程封面下载到本地。

不要长期只依赖外部图片链接。

这样即使数据源图片失效，本地仍可以正常显示。

------------------------------------------
10.2 本地上传
------------------------------------------

点击“更换封面”后，打开封面管理窗口。

提供：

- 上传本地图片
- 使用图片 URL
- 恢复默认封面
- 取消

支持常见图片格式：

- JPG
- JPEG
- PNG

二十七、关键交互流程
==================================================

------------------------------------------
流程一：首次打开
------------------------------------------

1. 用户打开网站。
2. 数据库为空。
3. 页面显示安静的空状态。
4. 用户点击“添加动漫”。
5. 打开搜索导入窗口。

------------------------------------------
流程二：搜索进击的巨人
------------------------------------------

1. 用户输入“进击的巨人”。
2. 系统请求已启用的数据源。
3. 展示真实搜索结果。
4. 尽量根据数据源关系分组。
5. 用户勾选第一季、第二季、第三季、最终季和 OVA。
6. 用户为选中作品选择“已看完”或“在看”。
7. 用户点击“导入选中作品”。
8. 系统获取资料并下载封面。
9. 成功导入的作品出现在首页海报墙。
10. 失败的条目显示具体原因。

------------------------------------------
流程三：查看详情
------------------------------------------

1. 用户点击首页中的动漫海报。
2. 打开居中详情模态窗口。
3. 首页保持在背景中。
4. 不在页面底部插入任何内容。
5. 模态窗口显示海报、标题、简介、状态和相关作品。
6. 用户关闭窗口后回到原来的海报位置。

------------------------------------------
流程四：修改状态
------------------------------------------

1. 用户打开详情模态窗口。
2. 将状态从“在看”修改为“已看完”。
3. 系统立即保存。
4. 首页对应卡片更新。
5. 左侧数量更新。
6. 当前筛选为“在看”时，该卡片从列表中消失。

------------------------------------------
流程五：更换封面
------------------------------------------

1. 用户打开详情模态窗口。
2. 点击“更换封面”。
3. 选择上传本地图片。
4. 查看预览。
5. 确认保存。
6. 首页和详情窗口立即显示新封面。
7. 原始数据源封面仍然保留。
8. 用户以后可以恢复默认封面。
