# 「我的动漫库」分轮执行提示词

本文件由原始一次性需求提示词按照 AGENT.md《通用执行规则》重构而成。
原始完整需求已原样保存至 REQUIREMENTS.md，本文件只引用其章节编号，不重复粘贴需求内容（规则 5）。

## 项目文件导航

| 文件 | 作用 | 产生时机 |
|---|---|---|
| AGENT.md | 通用执行规则，所有轮次必须遵守 | 已存在 |
| REQUIREMENTS.md | 原始完整需求，唯一需求事实来源 | 已存在 |
| Project-prompts.md | 本文件：分轮执行提示词 | 已存在 |
| FEATURE-BOUNDARIES.md | 功能白名单与黑名单 | 第 0 轮产出 |
| PROGRESS.md | 项目进度文档，每轮结束必须更新 | 第 0 轮产出 |
| ARCHITECTURE.md | 已确认的技术架构，确认后不得每轮重设 | 第 1 轮产出 |

## 使用方式

1. **每次只执行一轮。** 开始某一轮时，向执行者发送一句话即可：

   > 请阅读 AGENT.md、Project-prompts.md 中的《通用轮次协议》与《第 N 轮》章节，并严格执行第 N 轮。

   不要把 REQUIREMENTS.md 全文粘贴进对话，执行者自行按章节读取（规则 5、17）。

2. **一轮完成并通过验证后立即停止**，等待人工确认后再开始下一轮（规则 1、3）。

3. **任何时刻只允许一个 Agent / 模型修改本仓库**（规则 4）。

4. **切换模型时**，以 REQUIREMENTS.md、代码仓库和 PROGRESS.md 为事实来源，不依赖上一模型未保存的对话记忆（规则 16）。

5. **模型选择**（规则 14、15）：
   - 强推理模型：架构设计、数据库、安全逻辑、外部 API 接入、删除操作、疑难错误、最终审计
   - 快速 / 低成本模型：脚手架搭建、普通 CRUD、样式微调、简单表单、文档补充

6. **架构变更例外通道**（规则 6、7）：若某轮发现必须调整 ARCHITECTURE.md 中已确认的架构或数据库结构，先停止当前轮，在 PROGRESS.md 中写明修改原因、影响范围、迁移方式，经人工确认后作为单独一轮执行。无明确缺陷或需求变化时不得调整。

7. **依赖与框架升级**（规则 29、30）：不与任何功能轮混合。如需升级，单独开一轮执行并完成兼容性验证。

## 通用轮次协议

以下内容对每一轮都生效。

### 开场（规则 17、18）

开始工作前必须先读取：

1. REQUIREMENTS.md 中本轮「必读章节」列出的部分
2. PROGRESS.md 的最新记录（第 0 轮尚未创建时跳过）
3. ARCHITECTURE.md（第 1 轮之后）
4. 本轮涉及的现有代码
5. 最近一次测试结果

不得假设任何功能已经完成。必须通过实际文件、数据库迁移、接口响应或测试结果确认（规则 18）。

### 执行（规则 2、8、9、24-28）

- 只做本轮「允许范围」内的事，不得触碰「禁止范围」，不得擅自扩展任务（规则 2）。
- 不得添加 REQUIREMENTS.md 未明确列出的功能、按钮、字段、页面或假数据（规则 24；REQUIREMENTS.md 第四、二十五、三十三章）。
- 需求不明确时选择最小实现（规则 25）；多方案可行时选依赖更少、结构更清晰、更易维护的方案（规则 26）。
- 不进行与本轮目标无关的重构，不为统一风格修改无关文件（规则 27、28）。
- 测试失败时只修复与本轮直接相关的问题（规则 8）。
- 修改文件范围尽量集中在本轮涉及的模块（规则 9）。

### 收尾（规则 3、19-23、34）

1. 运行本轮「验证要求」列出的检查。不要求全量测试，但必须足以验证本轮修改（规则 21、22）。
2. 不得伪造任何命令输出、测试结果、构建结果或 API 响应（规则 19）。无法执行的验证必须标注「未验证」并说明原因（规则 20）。
3. 按以下模板向 PROGRESS.md 追加记录（规则 23）：

   ```
   ## 第 N 轮：<标题>（YYYY-MM-DD）
   - 已完成：
   - 修改的关键文件：
   - 执行的命令：
   - 测试结果：
   - 已知问题：
   - 未完成内容：
   - 下一轮建议：
   ```

4. 对照本轮「完成标准」逐项确认。只有功能真实可用、关键错误已处理、相关测试通过、文档已更新、没有超出需求范围、没有遗留未说明的核心 TODO 时，本轮才算完成（规则 33、34）。
5. 全部满足后立即停止，不得自动开始下一轮（规则 3）。
